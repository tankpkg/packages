# Reading EXPLAIN ANALYZE

Sources: Winand (SQL Performance Explained), PostgreSQL documentation, MySQL documentation

Covers: EXPLAIN output interpretation, node types, scan types, join types, cost model, bottleneck identification.

## Running EXPLAIN
The EXPLAIN command is the primary tool for understanding how a database engine intends to execute a query. While a standard EXPLAIN shows the planner's intended path based on statistics, it does not account for runtime realities like disk I/O contention, CPU spikes, or concurrent locks.

### EXPLAIN vs EXPLAIN ANALYZE
- **EXPLAIN**: Shows the execution plan generated by the query planner. It uses table statistics to estimate costs and row counts. It is safe to run on any query as it does not execute the code. Use this for quick checks on large tables where execution would take too long.
- **EXPLAIN ANALYZE**: Actually executes the query. In addition to the plan, it provides real timing data and actual row counts. This is essential for finding cases where the planner's estimates are wrong.
- **WARNING**: Because EXPLAIN ANALYZE executes the query, it will perform any data modifications (INSERT, UPDATE, DELETE) contained in the statement. To test writes safely, wrap the command in a transaction and roll it back.

```sql
-- Safe write testing pattern
BEGIN;
EXPLAIN ANALYZE DELETE FROM users WHERE last_login < '2020-01-01';
ROLLBACK;
```

### Recommended Syntax (PostgreSQL)
The default text output is often sufficient, but adding specific flags is critical for identifying I/O bottlenecks and detailed behavior.
- **ANALYZE**: Execute the query and show actual timings and row counts.
- **BUFFERS**: Show data on block hits, reads, dirty, and written. Essential for identifying I/O bound queries.
- **VERBOSE**: Show additional information such as the list of columns output by each node and schema-qualified names.
- **WAL**: Show Write Ahead Log activity (useful for evaluating write performance).
- **TIMING**: Includes the time spent in each node (on by default with ANALYZE).
- **SUMMARY**: Includes summary information (execution time) at the end of the report.
- **SETTINGS**: Includes information about configuration parameters that affect the plan.
- **FORMAT**: Can be TEXT, XML, JSON, or YAML. JSON is best for external visualization tools.

```sql
-- The gold standard for manual analysis
EXPLAIN (ANALYZE, BUFFERS, VERBOSE, SETTINGS) 
SELECT * FROM orders WHERE status = 'shipped';

-- For programmatic analysis or external tools
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) 
SELECT * FROM orders;
```

### Recommended Syntax (MySQL)
MySQL introduced a more detailed output format in version 8.0.18 and a tree format in 8.0.16.
```sql
-- Standard tabular output (legacy)
EXPLAIN SELECT * FROM orders WHERE id = 1;

-- Detailed execution timing (MySQL 8.0.18+)
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 5;

-- Tree format (easier to read flow, similar to Postgres)
EXPLAIN FORMAT=TREE SELECT * FROM orders;

-- Structured data for deep inspection
EXPLAIN FORMAT=JSON SELECT * FROM orders;
```

## Reading the Output (PostgreSQL)
PostgreSQL query plans are tree structures. In the text representation, indentation indicates the relationship between nodes.

### Direction of Flow
- **Innermost to Outermost**: The plan is executed from the bottom up (innermost indented nodes first).
- **Data Pipeline**: Each child node passes its result rows up to its parent node.
- **Execution Order**: A parent node may stop requesting rows from a child if its own condition is met (e.g., a LIMIT node). This is known as "early exit".

### Node Header Breakdown
A typical node header looks like this:
`Index Scan using idx_status on orders (cost=0.42..8.44 rows=10 width=42) (actual time=0.015..0.017 rows=12 loops=1)`

- **Type**: The operation performed (e.g., Index Scan, Seq Scan, Hash Join).
- **Cost (Startup..Total)**: 
  - **Startup cost**: Estimated work required before the first row can be returned (e.g., the time to build a hash table).
  - **Total cost**: Estimated work required to return all rows, assuming the node is run to completion.
- **Rows**: The planner's estimate of the number of rows this node will output.
- **Width**: Estimated average width in bytes of the rows returned by this node.
- **Actual Time (Startup..Total)**: 
  - Measured in milliseconds. 
  - Startup time is time to first row. Total time is the time taken to return the last row.
- **Actual Rows**: The actual number of rows returned during execution.
- **Loops**: How many times this specific node was executed.

### Understanding Buffers
When using the `BUFFERS` flag, you see details about page access:
- **Shared Hit**: Pages found in the PostgreSQL buffer cache (no disk read).
- **Shared Read**: Pages read from the OS cache or disk (I/O involved).
- **Shared Dirtied**: Pages modified by this query.
- **Shared Written**: Pages pushed to disk by this query.
- **Local/Temp Read/Hit**: Concerns temporary tables or files used for internal operations like large sorts or hash joins.

### Understanding Loops and Timing
If a node has `loops=1000`, the `actual time` and `actual rows` values are **per loop averages**. To find the total time spent in that node, you must multiply `actual time` by `loops`. Failing to do this is the most common mistake in interpreting EXPLAIN output.

## Parallel Query Execution
Modern databases can use multiple CPU cores to execute a single query.
- **Gather**: The node where the leader process collects results from worker processes.
- **Gather Merge**: Similar to Gather, but maintains the sort order from workers.
- **Parallel Seq Scan**: Multiple workers read different parts of the table simultaneously.
- **Parallel Hash Join**: Workers build or probe hash tables in parallel.
- **Workers Planned**: How many parallel processes the database intended to use.
- **Workers Launched**: How many processes it actually started.

### Parallel Scaling Issues
If a plan shows `Workers Planned: 4` but `Workers Launched: 0`, the system may be at its `max_parallel_workers` limit, forcing the query to run sequentially. This often happens on busy systems and results in much longer execution times than the planner anticipated.

## Comprehensive Node Types Reference
Beyond scans and joins, several other nodes are common in complex queries.

### Aggregate and Set Nodes
- **HashAggregate**: Uses a hash table to group rows and calculate aggregates. Fast for many groups if they fit in `work_mem`.
- **GroupAggregate**: Requires input to be sorted by the grouping key. Used for very large datasets or when indexes already provide sorting.
- **MixedAggregate**: Used for complex grouping sets (`ROLLUP`, `CUBE`).
- **HashSetOp**: Implements `INTERSECT` or `EXCEPT` via hash table.
- **Recursive Union**: Implements `WITH RECURSIVE` queries using a working table.

### Join Nodes
- **Nested Loop**: O(N*M) complexity. Best for small outer, indexed inner.
- **Hash Join**: O(N+M) complexity. Best for large, unsorted equality joins.
- **Merge Join**: O(N log N + M log M) complexity. Best for sorted data or non-equality joins.

### Utility Nodes
- **Sort**: Reorders rows. Spills to disk (External Merge) if `work_mem` is exceeded.
- **Unique**: Removes duplicates; requires sorted input.
- **Limit**: Stops execution after N rows. Highly efficient for early exits.
- **Materialize**: Caches results in memory to avoid re-executing a child node (e.g., in a nested loop).
- **WindowAgg**: Processes window functions like `ROW_NUMBER()` or `RANK()`.
- **Result**: Evaluates constant expressions or returns a single row.
- **Append**: Combines results from multiple children (e.g., `UNION ALL`).
- **Merge Append**: Combines sorted child results while maintaining order.
- **WorkTable Scan**: Scans the temporary table used in recursive queries.
- **Values Scan**: Scans a constant set of rows provided in a `VALUES` clause.
- **CTE Scan**: Scans the output of a Common Table Expression.

## Scan Types Deep Dive
Scans determine how the database retrieves rows from a single table.

### Sequential Scan (Seq Scan)
The database reads every page of the table in physical order. 
- **Pros**: Efficient for reading 10%+ of a table; sequential I/O is faster than random I/O.
- **Cons**: Linear time O(N); reads irrelevant data.

### Index Scan
The database uses a B-tree index to find pointers (TIDs) to the rows, then visits the heap to retrieve the row data. Each visit to the heap is a random I/O operation. If many rows are returned, the cost of random I/O can exceed the cost of a sequential scan.

### Index Only Scan
If the index contains all the columns needed by the query, the database can retrieve everything from the index pages. This is the fastest scan type but requires the Visibility Map to be up to date. If it's not, "Heap Fetches" will be performed, degrading performance.

### Bitmap Index Scan and Bitmap Heap Scan
1. **Bitmap Index Scan**: Finds all matching rows in the index and builds a bitmap of the pages they occupy.
2. **Bitmap Heap Scan**: Reads the pages in the bitmap in physical order.
This strategy is used when an Index Scan would perform too many random I/O operations or when combining multiple indexes.

## Cost Model and Calculations
PostgreSQL uses unitless cost estimates to compare plans.

### Fundamental Constants
- **seq_page_cost (1.0)**: The cost to read one page sequentially.
- **random_page_cost (4.0 default)**: The cost to read one page via random access. On SSDs, set this to 1.1.
- **cpu_tuple_cost (0.01)**: The cost to process one row.
- **cpu_index_tuple_cost (0.005)**: The cost to process one index entry.
- **cpu_operator_cost (0.0025)**: The cost to perform a comparison.

### Calculation Example: Seq Scan
For a table with 10,000 rows across 100 pages:
- I/O Cost: `100 pages * seq_page_cost(1.0) = 100.0`
- CPU Cost: `10,000 rows * cpu_tuple_cost(0.01) = 100.0`
- **Total Cost**: `200.0`

### Impact of Memory Config
- **effective_cache_size**: A hint to the planner about how much data is likely to be cached in memory. Larger values favor index scans.
- **work_mem**: Buffer for sorts and hash joins. Spilling to disk (External Merge Sort) is often 10-100x slower.

## Red Flags in Query Plans
Identifying these patterns is the first step in query optimization.

| Red Flag | Description | Typical Fix |
| :--- | :--- | :--- |
| **Seq Scan on Large Table** | Table scan exceeding 10,000 rows. | Add an index or refine filter. |
| **Major Estimation Error** | `rows=1` but `actual rows=100,000`. | Run `ANALYZE` or check statistics. |
| **External Merge Sort** | Sort spilled to disk. | Increase `work_mem` or add index. |
| **Hash Join Spill** | Hash table too large for memory. | Increase `work_mem`. |
| **High Buffers: Read** | Data not in cache, forcing disk I/O. | Increase cache or reduce data scanned. |
| **Filtering at Node Level** | Filter applied after full scan. | Move filter to an index (`Index Cond`). |
| **High Loop Count** | Nested loops on unindexed inner nodes. | Add index to the join key. |
| **Materialize Node** | Intermediate result stored in memory. | Check if join order can be improved. |

## Estimation Errors and Statistics
The query planner depends on the accuracy of the statistics catalog.

### Why Estimates Fail
1. **Stale Statistics**: `ANALYZE` hasn't run after major data changes.
2. **Correlated Columns**: e.g., `make='Toyota'` and `model='Camry'`. The planner thinks these are independent.
3. **Complex Expressions**: Filters like `WHERE upper(name) = 'BOB'` are hard to estimate.
4. **Data Skew**: Some values are much more common than others.

### Improving Estimates
- **Statistics Target**: `ALTER TABLE users ALTER COLUMN email SET STATISTICS 1000;`
- **Extended Statistics**: `CREATE STATISTICS s1 (dependencies) ON city, zip FROM users;`
- **Functional Indexes**: Index the expression itself: `CREATE INDEX ON users (lower(email));`

## MySQL EXPLAIN Differences
MySQL 8.0 introduced hash joins and a more sophisticated planner.

### Access Types (type column)
1. **const/system**: Fastest, single row lookup via PK/Unique.
2. **eq_ref**: Single row lookup for each row from previous table (Unique join).
3. **ref**: Non-unique index lookup.
4. **range**: Index range search (e.g., `BETWEEN`, `>`, `<`).
5. **index**: Full index scan.
6. **ALL**: Full table scan (Worst).

### Key Indicators
- **key_len**: Length of the index used.
- **Extra: Using filesort**: Indicates a sort that cannot be fulfilled by an index.
- **Extra: Using temporary**: Indicates a temporary table was created.

## Common Performance Patterns
### 1. The Small Table Scan
A `Seq Scan` on a table with 50 rows is not a bug. Random I/O setup for an index scan is often slower than a simple sequential read.

### 2. The Early Exit
A `LIMIT 1` query might show a high `total cost` but a very low `actual time` because the database stopped early.

### 3. Nested Loop Death Spiral
If the inner side of a nested loop is a `Seq Scan`, performance is O(N*M). Adding an index usually fixes this.

### 4. Visibility Map Stale
`Index Only Scan` with high `Heap Fetches` means the visibility map is not up to date. Run `VACUUM ANALYZE`.

## Detailed Annotated Examples
### Example 1: Postgres Index Only Scan
```
Index Only Scan using users_email_idx on users (cost=0.42..8.44 rows=1 width=32) (actual time=0.015..0.015 rows=1 loops=1)
  Index Cond: (email = 'user@example.com'::text)
  Heap Fetches: 0
  Buffers: shared hit=4
Planning Time: 0.123 ms
Execution Time: 0.045 ms
```
- **Analysis**: Perfect query. 4 buffer hits (index pages), 0 heap fetches. Total execution 45 microseconds.

### Example 2: Postgres Disk Spilling Sort
```
Sort  (cost=1234.56..1250.00 rows=1000 width=128) (actual time=500.0..550.0 rows=1000 loops=1)
  Sort Key: created_at DESC
  Sort Method: external merge  Disk: 5000kB
  Buffers: shared hit=1000, temp read=625, temp written=625
  ->  Seq Scan on large_table  (cost=0.00..500.00 rows=10000 width=128) (actual time=0.01..50.0 rows=10000 loops=1)
```
- **Analysis**: The sort spilled to disk because it needed 5MB but `work_mem` was likely 4MB or less. Notice `temp read` and `temp written`. Fix by increasing `work_mem`.

### Example 3: MySQL 8.0 Format=Tree
```
-> Limit: 1 row(s)  (cost=0.70 rows=1) (actual time=0.123..0.124 rows=1 loops=1)
    -> Nested loop inner join  (cost=0.70 rows=1) (actual time=0.122..0.123 rows=1 loops=1)
        -> Index lookup on u using users_email_idx (email='test@test.com')  (cost=0.35 rows=1) (actual time=0.080..0.081 rows=1 loops=1)
        -> Single-row index lookup on o using PRIMARY (id=u.last_order_id)  (cost=0.35 rows=1) (actual time=0.040..0.041 rows=1 loops=1)
```
- **Analysis**: High performance join. Index lookup on users followed by a single-row primary key lookup on orders.

## Glossary of Terms
- **Heap**: The physical storage for table rows.
- **TID**: Physical address of a row (Page + Offset).
- **Selectivity**: Probability that a row matches a filter.
- **Optimization Fence**: Operation that prevents the planner from optimizing across it.
- **Visibility Map**: Tracks pages with only visible rows.
- **Statistics Target**: Number of entries in the histogram used for estimation.
- **Buffer Cache**: In-memory area where database pages are cached.

## Optimization Checklist
- [ ] Is `ANALYZE` up to date?
- [ ] Are estimated rows close to actual rows?
- [ ] Are any large tables doing a `Seq Scan`?
- [ ] Are there any disk-spilling `Sort` or `Hash Join` nodes?
- [ ] Is the join order correct (smaller table on the outer side for Nested Loops)?
- [ ] Are `Index Only Scans` doing high `Heap Fetches`?
- [ ] Is the query parallelized if it's large?
- [ ] Are filters happening at the index level (`Index Cond`)?

## Frequently Asked Questions
### Q: Why is my Index Scan slower than a Seq Scan?
A: An Index Scan performs random I/O for every row retrieved. If the query returns many rows, the cost of these random seek operations exceeds the cost of a continuous sequential read of the whole table.

### Q: What is a "Recheck Cond"?
A: In a Bitmap Heap Scan, if the bitmap becomes too large (lossy bitmap), the database only remembers which pages contain matching rows, not the exact rows. It must "recheck" the condition for every row on that page.

### Q: Why does my plan change when I add a LIMIT?
A: The planner optimizes for the total time to get the *requested* rows. With `LIMIT 10`, it may choose a slow overall plan that returns the first 10 rows very quickly (e.g., using an index for ordering instead of a hash join).

## EXPLAIN Workflow
1. **Get the Baseline**: Run `EXPLAIN (ANALYZE, BUFFERS) query;`.
2. **Find the Bottleneck**: Identify the node with the highest `actual time`. Remember to multiply by `loops`.
3. **Verify Estimates**: Compare `rows` to `actual rows`. Run `ANALYZE` if off.
4. **Check I/O**: Look for `shared read` in the Buffers section.
5. **Evaluate Scan Types**: If a large table uses `Seq Scan`, can you add an index?
6. **Evaluate Join Types**: If a `Nested Loop` is slow, check if the inner join key is indexed.
7. **Check Memory**: If you see `Disk` sorts or `Batch` hash joins, increase `work_mem`.
8. **Check Filter Placement**: Ensure filters happen at the index level.
9. **Iterate**: Apply one change at a time and re-test.
10. **Verify**: Ensure build passes and no regressions occur.
